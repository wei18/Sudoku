## 設計決定 (Design decisions)

- **cellSide formula (non-square fit)** — Sudoku's `BoardView` uses `side/dimension`
  on a SQUARE board. MS boards are non-square (Expert 16×30). Fit by BOTH axes so the
  whole board fits the offered rect: `cellSide = floor(min(availW/cols, availH/rows))`.
  `floor` keeps the glyph crisp on integer point boundaries (footgun: fractional cell
  sizes blur text on the board). availW/availH = offered size minus inter-cell spacing.

- **Min-cell floor = 32pt → scroll, not shrink** — HIG tap-target min is 44pt but the
  board is a dense grid (Apple itself ships sub-44 grid cells; the project's Sudoku 9×9
  also shrinks below 44 on small iPhones). Picked **32pt** as the floor: comfortable
  enough for a deliberate reveal tap, and a flag tap is now mode-driven (no precision
  long-press needed). When `floor(min(...))` would drop below 32, clamp `cellSide` to 32
  and wrap the grid in a `ScrollView([.horizontal, .vertical])` so the board scrolls
  instead of shrinking into un-tappable territory. Below the floor we DON'T center —
  scroll content sizes to `cols*cellSide × rows*cellSide`.

- **Font derived from cellSide** — `cellSide * 0.55` (spec). Replaces literal `size: 16`.
  Footgun note: grids normally keep a FIXED font, but here the cell itself scales, so the
  glyph must track it to stay proportional. We do NOT let Dynamic Type re-scale the board
  glyph (`.font(.system(size:))` is fixed-size, not a text style) — keeps the grid stable.

- **Interaction mode toggle** — modeled on DigitPad's pencil toggle. Added a
  `@State private var interactionMode: InteractionMode` (`.reveal` / `.flag`) local to the
  view (no VM change needed — it's pure view interaction state, like Sudoku's pencil mode
  which IS on the VM, but MS VM is MVP and mode has no engine semantics, so view-local is
  surgical). Two-segment control above the board. In `.flag` mode a tap calls
  `onToggleFlag`; in `.reveal` mode a tap calls `onReveal`. The button's primary action is
  now mode-dependent.

- **Accelerators preserved** — iOS long-press still flags, macOS right-click contextMenu
  still flags, regardless of mode. They are accelerators on top of the mode toggle.

## 折衷 (Tradeoffs)

- **View-local @State vs VM-held mode** — Sudoku holds pencilMode on the VM because it
  affects undo/note semantics. MS flag-mode has zero engine impact (it just routes which
  action a tap fires). Putting it on the VM would be speculative coupling (Karpathy §2).
  Chose view-local `@State`. If a future Tier-1 needs to persist mode, promote then.

- **Scroll axes** — Could scroll only horizontally (Expert is wide, not tall on iPhone
  portrait: 30 cols × 32 = 960 wide, 16 rows × 32 = 512 tall fits most phones vertically).
  But Intermediate is 16×16 and landscape/short windows can clip vertically too. Used
  both axes for robustness; when content fits, ScrollView is inert (no visible scrolling).

## 未決 (Open questions)

- None load-bearing. The 32pt floor is a judgement call (documented above); Designer
