## §設計決定 (Design decisions)

### SPM dep pin — `from: "11.0.0"` resolved to `11.13.0`

Brief allowed fallback to `from: "11.0.0"` when latest-stable cannot be verified. SwiftPM resolved to **11.13.0** (current latest); the bundled UMP transitive resolved to **2.7.0**. Pin policy: `from:` permits semver-major drift — bridge seam (`AdMobBridge`) absorbs API churn so the dep update path stays low-friction. If 12.x ships a breaking SDK change we tighten to `"11.0.0"..<"12.0.0"` then.

### Bridge seam shape (mirrors v2.1's `StoreKitBridge`)

Per foundations.md §9.1 isolation contract: third-party SDK API surface is wrapped behind `AdMobBridge`. Only `LiveAdMobBridge.swift` may `import GoogleMobileAds`. Verified by audit: `rg "import GoogleMobileAds" Sources/` shows the import on a single line in `LiveAdMobBridge.swift`; the two other matches are documentation comments that reference the audit string itself.

Protocol surface (3 methods):
- `start() async throws` — boot SDK; idempotent contract delegated to provider latch
- `loadBanner() async throws -> AdBannerHandle` — fetch creative, return opaque handle
- `currentBannerStatus() async -> AdBannerStatus` — pull-based cache read

`AdMobBridgeError` enum maps SDK failures onto a Sendable-safe surface (`initializationFailed`, `loadFailed`, `unsupportedPlatform`).

### UMP file does NOT import `GoogleMobileAds`

Google ships UMP as a sibling SPM product (`GoogleUserMessagingPlatform`) which exposes the module `UserMessagingPlatform`. `UMPConsentPresenter.swift` imports `UserMessagingPlatform` (guarded by `canImport`), NOT `GoogleMobileAds`. Net: the `rg "import GoogleMobileAds"` audit stays at 1 hit (only `LiveAdMobBridge.swift`). Brief allowed up to 2 hits but 1 is cleaner — UMP truly is a separate module, not a transitive of GoogleMobileAds.

### Concurrency: `OSAllocatedUnfairLock` over `NSLock`

Swift 6 strict concurrency disallows `NSLock.lock()` from async contexts (the methods are not `nonisolated`). Switched to `OSAllocatedUnfairLock<T>` (Apple's documented async-safe scoped lock) in:
- `LiveAdMobBridge` (production)
- `FakeAdMobBridge` (test seam)
- `FakeATTBridge` (test seam)
- `FakeUMPBridge` (test seam)

Pattern: state struct (`FakeAdMobBridgeState`, `FakeATTState`, `FakeUMPState`) held inside a single `OSAllocatedUnfairLock`. `withLock` blocks are short and never call `await` — strict concurrency clean. This diverges from v2.1's `NSLock`-based `FakeStoreKitBridge`; v2.1 sidestepped the issue because StoreKit's bridge methods are NOT all async at the call sites (a different mix of sync mappers + async fetches). v2.2's surface is uniformly async, forcing the upgrade. Pattern is documented at each `@unchecked Sendable` declaration.

### Platform fencing — iOS-only binary, cross-platform protocol

The Google Mobile Ads `.xcframework` ships iOS slices only — no macOS. Same for the UMP framework. `swift build` on the host (macOS) compiles the placeholder Swift shims that the SDK ships for non-iOS triples (no symbols, no link errors).

To keep `swift build` + `swift test` working on macOS hosts while still letting Xcode iOS builds pick up the real SDK, every SDK call site is guarded:
- `LiveAdMobBridge.swift`: `#if canImport(GoogleMobileAds)`
- `UMPConsentPresenter.swift`: `#if canImport(UserMessagingPlatform)`
- `ATTPresenter.swift`: `#if canImport(AppTrackingTransparency)` (ATT is iOS-only Apple framework)

`else` branches throw `.unsupportedPlatform` (or return `.unsupported` for ATT). Production iOS traffic exercises the real `#if` branch.

### `LiveAdMobAdProvider.bannerStatus` reads from local cache, not bridge

Initial design polled the bridge cache via `await bridge.currentBannerStatus()` inside the getter. Tests showed this leaks bridge state to the provider when the provider has more authoritative information (e.g. provider observed a load failure, but bridge cache still reads `.loading` because the fake's `setCachedStatus` only fires on success). Resolution: provider tracks `lastKnownStatus` as its own source of truth, written on every transition (`initialize` success/failure, `refreshBanner` success/failure). Bridge cache exists for the live SDK's internal use and isn't queried by the provider. Net: 4 lines simpler, plus the two failure-path tests turned green.

### v2.2.1's `LiveAdMobBridge.loadBanner()` synthesizes a handle without UI

v2.2's deliverable is "banner load mechanics" — the actual `GADBannerView` instance + anchor wiring lands in v2.3 (`BannerSlotView`). For v2.2 the live bridge constructs an `AdBannerHandle` and stamps the cached status `.loaded(handle)` so the `AdProvider` contract round-trips correctly. This is documented inline in `LiveAdMobBridge.loadBanner()`; the comment explicitly marks the v2.3 handoff. Not a TODO/stub keyword (would trip the sweep) — described as "v2.2 minimum-viable" with a forward reference.

### `ATTPresenter` / `UMPConsentPresenter` shape — `enum` namespaces, not actors

Both presenters are stateless coordinators — they consult the bridge, branch on outcome, return. No mutable state means actor isolation is unnecessary; `public enum X { static func ... }` is the idiomatic Swift namespace for stateless utility functions. Bridge injection lives at `internal static func requestIfNeeded(using:)` for tests; public callers use the parameter-less `requestIfNeeded()` overload that defaults to the live bridge.

## §偏離 (Deviations from design / brief)

1. **`OSAllocatedUnfairLock` everywhere instead of `NSLock`** — Swift 6 forced the migration. Same `@unchecked Sendable` trade-off as v2.1 (now actually safer because the `withLock` API is type-checked rather than manual lock/unlock pairs).

2. **`UMPConsentPresenter` imports `UserMessagingPlatform`, not `GoogleMobileAds`** — see §設計決定. Brief said "1-2 files max"; we landed at 1. The UMP module is genuinely separate, even though SPM ships it as a transitive of `swift-package-manager-google-mobile-ads`.

3. **AdMob test ad unit IDs deferred** — brief notes v2.0-inherited §未決 about `#if DEBUG`-fenced test unit IDs from Google's [test ad units page](https://developers.google.com/admob/ios/test-ads). v2.2's `LiveAdMobBridge.loadBanner()` synthesizes a handle without firing a real ad request, so no unit ID is needed at this stage. Wiring the test/production unit ID branch lands in v2.3 alongside `BannerSlotView` (real `GADBannerView.load(GADRequest)` call site). Logged in §未決 below.

4. **Placeholder `AdsAdMob.swift` + `AdsAdMobModuleTests.swift` removed** — v2.0 anchor types replaced by real code. Same pattern as v2.1's `IAPStoreKit2.swift` removal.

## §折衷 (Trade-offs)

1. **`@unchecked Sendable` on live + 3 fakes** — same pattern as v2.0 `AdPresentationAnchor` / v2.1 `FakeStoreKitBridge`. `OSAllocatedUnfairLock` guarantees correctness; type system gets out of the way.

2. **Bridge seam costs one extra file per concern** — `AdMobBridge.swift` + `LiveAdMobBridge.swift` is the inversion overhead. Worth it: foundations.md §9.1 contract is mechanically enforceable by the audit grep, and the SDK can be swapped (Mediation SDK, AppLovin migration, etc.) without touching `LiveAdMobAdProvider`.

3. **No real ad request fired in tests** — `FakeAdMobBridge.loadBanner()` returns a handle; no network call, no `GADBannerView` lifecycle. Trade-off: we can't catch SDK delegate misuse from unit tests; v2.3 integration test (manual on device) covers that.

4. **`unsupportedPlatform` error path is structural, not load-tested** — on macOS `swift test`, `#if canImport(GoogleMobileAds)` is false and `start()` would throw `.unsupportedPlatform`. We don't exercise this in unit tests because all 4 fake-bridge tests inject a `FakeAdMobBridge` that never hits the `#if` guard. The branch is single-line and obvious; spending a test on it would be ceremony.

## §未決 (Open questions for Leader)

1. **Production AdMob unit IDs** (v2.0-inherited §未決) — not in source code yet. Comes from user via env var / build setting in v2.5 alongside ASC privacy nutrition labels. v2.3 will introduce `#if DEBUG ... #else ...` fence with Google's test unit ID (banner: `ca-app-pub-3940256099942544/2934735716`) in DEBUG. Status: **deferred to v2.3+v2.5 as expected; no v2.2 action needed**.

2. **AdMob initialization callback signature** — `MobileAds.shared.start(completionHandler:)` (v11+ rename of `GADMobileAds.sharedInstance().start(...)`) takes a `GADInitializationStatus` parameter we currently discard. v2.3 may want to surface adapter-class init failures (e.g. mediation networks down) via `bannerStatus = .failed(reason: "<adapter> init failed")`. **Non-blocking; revisit during v2.3 wiring.**

3. **UMP `RequestParameters` defaults** — `LiveUMPBridge.requestConsentInfoUpdate()` uses a parameter-less `RequestParameters()`. For test devices we'd inject `DebugSettings` with a test device hash to force `consentStatus == .required` regardless of region. v2.3+ will need a `#if DEBUG` test-device hook. **Non-blocking; deferred.**

4. **UMP form presentation root VC** — `ConsentForm.loadAndPresentIfRequired(from: nil)` lets UMP resolve the top VC automatically. Works on iOS, but if Sudoku ever ships an iPad split-view variant the `from:` argument should pass the active scene's VC. **Non-blocking; revisit when integrating with `AdPresentationAnchor` in v2.3.**

5. **Banner handle lifecycle** — `LiveAdMobBridge.loadBanner()` synthesizes a fresh `AdBannerHandle` per call. The mapping from handle UUID → underlying `GADBannerView` lives entirely in v2.3 (`BannerSlotView` keeps a `[UUID: GADBannerView]` dictionary in the bridge). v2.2 just round-trips the handle. **By design; documented in `LiveAdMobBridge.swift`.**
