Author: Developer (Claude Opus 4.8) → for Leader review

## TL;DR / design-fork decision

Issue #331 asks for a shared **SettingsKit** consolidating the Settings
surface across both games. Step-0 assessment found that this framework is
**already ~80% built inside `GameShellKit/GameShellUI`** through prior
incremental PRs (X4, #277, #287, #321). The issue's own `Shape (likely)`
section literally describes what `GameShellUI` already is.

**Fork taken:** do NOT create a new `SettingsKit` package. Instead, ship the
one genuinely-missing #331 piece — the **Notices / 宣告 section** — as a new
shared component in `GameShellUI`, alongside its siblings
(`SettingsShellView`, `SettingsAboutStorage`, the reminder primer). This is the
surgical / simplicity-first choice; a new package would re-home already-shared,
tested code for zero functional gain.

The broader "feature-toggle declarative system" the issue speculates about is
NOT built — see §Deferred. That is the real fork the Leader should rule on.

## What this PR builds

`GameShellUI/SettingsNoticesSection.swift` (new shared target file):
- `SettingsNoticesConfig` — a `Sendable` value the host constructs
  (acknowledgements action, privacy/support URLs, copyright). App-injected at
  the composition root.
- `SettingsNoticesSection` — `Section("Notices")` with optional rows:
  Acknowledgements (host action), Privacy Policy + Support (`Link`),
  Copyright. Theme-decoupled via `tintColor:` (mirrors `SettingsAboutVersionRow`).
- No monetization / UserNotifications / LicensePlist import leaks into
  GameShellUI — the acknowledgements destination is a host closure.

Wiring (both apps):
- Sudoku: `SettingsView` gains `notices:` param → `RouteFactory` threads
  `settingsNotices` → `AppComposition.live()` builds it (acknowledgements
  deep-links to the app's iOS Settings page where the Settings.bundle
  surfaces; copyright derived; **privacy/support URLs intentionally unwired**).
- Minesweeper: `SettingsView` gains `notices:` param → `LiveRouteFactory`
  `.settings` builds the identical config inline.

Tests: 3 new in `GameShellUITests/SettingsNoticesSectionTests`, +1 each in the
Sudoku and MS SettingsView test files. Localization: 4 new keys (Notices,
Acknowledgements, Privacy Policy, Support) × 7 locales added to BOTH app
catalogs; `gen:l10n_fixture` regenerated; `scan:l10n` gate green.

## Flagged for Leader — open decisions

1. **Privacy Policy + Support URLs are unwired** (`nil` in both apps). There is
   no canonical public URL committed anywhere in the repo (only ASC metadata's
   `privacy_policy_url` field). Wiring fabricated URLs would be a guess. The
   `SettingsNoticesConfig` fields exist and are optional — once the user
   provides the canonical URLs, wiring is a one-line change at each
   composition root. **Decision needed: supply the URLs.**

2. **Copyright string** is hardcoded `"© <year> Wei"`. Confirm the holder name
   / format (and whether it should be localized — currently rendered verbatim,
   no translation, which is conventional for a copyright line).

3. **Acknowledgements deep-link target.** iOS opens the app's own Settings page
   (`UIApplication.openSettingsURLString`) where the LicensePlist
   `Settings.bundle` lands. macOS has no equivalent → the row is omitted on
   macOS. Alternative (future): an in-app acknowledgements list view instead of
   the system Settings.bundle round-trip. Not built; flag if desired.

## Deferred (the real "new framework" the issue speculates about)

The issue's `Feature toggles` bullet — *"a declarative feature-flag/toggle
system (persisted, mutable), the home for reminders on/off, music/SFX (#330),
CaptureGuard kill-switch (#286)"* — is **NOT built here.** Reasons:
- Its inputs are mostly parked: #330 (audio) and #286 (CaptureGuard) are
  backlog/parked; #321 (reminder time) already has a bespoke, working store
  (`ReminderSettingsStore`).
- A declarative toggle system with a persistence-policy matrix
  (UserDefaults vs CloudKit per setting) is a sizable design with no current
  consumer demanding it. Building it now would be speculative (karpathy
  §Simplicity / §Surgical).

Recommendation: keep #331 open scoped to "feature-toggle system", land this
Notices slice under it (or a sub-issue), and design the toggle system when its
first real consumer (#330 audio) is unparked.
