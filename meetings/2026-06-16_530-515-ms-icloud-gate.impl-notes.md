## 設計決定 (Design decisions)

- **#530 two-phase shape** — MS `bootstrap()` already has graceful-degrade catch blocks for both `completed` and `failed` fetches, but both happen BEFORE `state = .loaded(...)`. Fix: immediately set `state = .loaded(mergeCards(trio, completed: [], failed: []))` right after `let trio = provider.dailyTrio(date:)` (synchronous, non-throwing), then call a new async `fillCompletionAndFailureOverlay(trio:date:)` Phase-2 function. The Phase-2 function does the same two fetches with the same catch logic, then re-merges and re-sets state — guarded by `guard case .loaded = state else { return }` defensive check (Sonnet's #526 nit).

- **#515 error classification** — `UserFacingError.classify` already maps `CKError.notAuthenticated` (code 9) and `accountTemporarilyUnavailable` (code 25) to `.iCloudSignedOut`. The gateway layer does NOT translate these to `PersistenceError` (only `.serverRecordChanged` → `.syncConflict` is translated). So a signed-out fetch propagates as a raw `CKError` with domain `"CKErrorDomain"` code 9. The fix catches via `UserFacingError.classify(error) == .iCloudSignedOut` rather than any specific type — this is exactly the helper that exists in the codebase and handles both CKError and PersistenceError.iCloudNotSignedIn.

- **#515 what to catch** — Docstring at L150-156 says schema/decode errors MUST propagate. The fix uses `if UserFacingError.classify(error) == .iCloudSignedOut { return nil }` in both `loadInProgress` and `latestInProgress`. All other errors (schema too new, decode failures, other CKErrors) are rethrown.

- **Test placement** — #530 offline tests: 190-line `MinesweeperDailyHubViewTests.swift` + ~130 lines of new tests stays under 400. Use new `MinesweeperDailyHubViewModelOfflineTests.swift` sibling to match Sudoku pattern (DailyHubViewModelOfflineTests.swift mirrors the split naming). #515 tests: 301-line `MinesweeperSavedGameStoreTests.swift` + ~80 lines would hit ~381, still under 400 — add in-place.

## 偏離 (Deviations)

- **Phase-2 function name** — Called `fillCompletionAndFailureOverlay(trio:date:)` (not just `fillCompletionOverlay` as in Sudoku) because MS has TWO overlay sources (completed + failed). Reflects the dual-fetch shape accurately.

- **`latestInProgress` also gets the iCloud-signed-out guard** — Issue #515 text specifically mentions `loadInProgress(recordName:)` (~L157) AND `latestInProgress()` (~L123). Both call `gateway.fetch`/`gateway.query` which propagate CKError.notAuthenticated. Both get the nil-return guard.

## 折衷 (Tradeoffs)

- **Catch `UserFacingError.classify == .iCloudSignedOut` vs catch typed PersistenceError** — The gateway only translates `.serverRecordChanged → syncConflict`; not-authenticated CKErrors propagate raw. Could add gateway translation, but that's bigger scope + different layer. Using `UserFacingError.classify` is the established pattern in this codebase (referenced in docstrings). Chosen approach is minimal surgical change.
