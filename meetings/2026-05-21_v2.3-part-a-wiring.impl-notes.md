## §設計決定 (Design decisions)

### MonetizationStateStore is a `typealias` to `AdGateStateStore`

The brief offered two shapes: (a) re-declare a protocol that inherits from `AdGateStateStore`, or (b) use the same protocol under a Persistence-local alias. Chose (b) — a `public typealias MonetizationStateStore = AdGateStateStore` — because:

- `AdGate` already accepts `any AdGateStateStore`. A separate protocol would require either a wrapper actor (extra layer + extra `await` hop) or `AdGateStateStore` to refine the new protocol (which forces every `MonetizationTesting` fake to also declare conformance to the Persistence-side type, leaking Persistence onto the test seam).
- The alias keeps Persistence as the place where "the live CloudKit-backed store lives" without inventing a new vocabulary. Callers in `AppComposition.live` write `let store = persistence.monetizationStateStore()` and feed it into `AdGate(store:)` — the same type the Fake satisfies in `.preview()` / `.tests()`.

### Record schema — `MonetizationState` singleton in `userZone`

- `recordType: "MonetizationState"`
- `recordName: "monetization-state"` — a fixed string. Only one such record per private DB (per iCloud account), matching the conceptual model of "this user's banner-frequency state".
- Lives in `com.wei18.sudoku.userZone` (the existing zone provisioned by `LivePrivateCKGateway`). No new zone needed.
- Schema auto-creates on first write — `LivePrivateCKGateway` is already in development-schema-write mode.

### Booleans encode as `Int` (`0` / `1`)

`RecordValue` (the package-internal value bag for CloudKit fields) intentionally restricts to `.string / .int / .date / .data / .stringSet`. There is no `.bool` case (foundations.md §1 keeps the protocol surface small). `hasPurchasedRemoveAds` therefore encodes as `.int(0)` / `.int(1)` with the read-side mapping `value != 0 → true`. This matches the pattern already used elsewhere in the package for boolean-ish fields.

### First-launch seeding writes through immediately

If `loadState()` finds no record, the store synthesises `AdGateState(firstLaunchAt: clock())` AND `gateway.save(...)` it before returning. Two reasons:

1. The 7-day grace-period math reads `firstLaunchAt` as an anchor; if we returned a fresh `Date()` every cold launch without persisting it, the grace would reset forever and the player would never see an ad.
2. CloudKit Private sync propagates the seed to other devices on the same iCloud account; subsequent cold launches on Device B read the same anchor (so banner timing is consistent cross-device, not per-install).

The seed is `await`-ed inside `loadState()`, so the call is not silent — failures bubble up to `AdGate`'s catch-and-suppress.

### Lazy gateway provider — `AppComposition.live()` callable from unit tests

Initial wiring had `LivePersistence.monetizationStateStore()` immediately read `gateway()`, which constructs `LivePrivateCKGateway()` → `CKContainer.default()`. That call traps with `CKException — containerIdentifier can not be nil` outside a properly entitled app bundle. Existing `LivePersistence` sidesteps this by deferring its gateway construction until `bootstrap()` or the first store-method call.

Replicated the same pattern: `LiveMonetizationStateStore` gained a second initializer that takes a `@Sendable () -> any PrivateCKGateway` closure; `LivePersistence.monetizationStateStore()` uses that overload so the gateway is materialized lazily on the first `loadState()` / `saveState()` call. The "happy path" initializer that takes a concrete gateway is still public and used by tests.

Net: `swift test` now runs `CompositionTests.liveCompositionWiresAllProtocols()` without crashing in CloudKit's internal `dispatch_once` block. Live IO behavior is unchanged because every IO call still routes through the same `LivePrivateCKGateway`.

### RouteFactory returns `AnyView`, not an associated-type Destination

`RouteFactory.view(for:) -> AnyView` was the load-bearing choice for this refactor. The alternative — `protocol RouteFactory { associatedtype Destination: View; func view(for: AppRoute) -> Destination }` — forces `RootView` and every test fixture to be generic over the factory's `Destination`, which propagates through `NavigationStackHost.destination` and breaks the existing `View+VM` ergonomics. `AnyView` pays a small SwiftUI identity-erasure cost but lets us store the factory as `any RouteFactory` in `AppComposition`, and the `NavigationStackHost`'s `destination` closure already wraps in `_ConditionalContent` / `AnyView` shapes from SwiftUI's `navigationDestination(for:)` API. The cost was already absorbed.

### Sidebar / Home-card modal callbacks bypass RouteFactory

The brief flagged an §未決: "does sidebar / HomeView present-native-modal action source go through RouteFactory or direct AppComposition access?". Inspected the actual call sites: the only modal-present in `RootView.sidebarPlaceholder` is `GameCenterDashboard.present()` — a static method with NO injected protocol deps. It doesn't need `adProvider` / `iapClient` / `adGate` to run.

Resolution: `GameCenterDashboard.present()` stays a `static` call. `RouteFactory` only handles route-pushed Views. v2.3.4-6 (HomeView banner card, Settings IAP rows) will read `composition.adProvider` / `iapClient` / `adGate` directly inside those Views — the deps are stored on `AppComposition` precisely for this case. RootView itself never sees them, so its 2-arg init is preserved.

This decision is recorded inline in `AppComposition.swift` (the property comment) and the §未決 in this file's resolved section.

### Persistence target gains a `MonetizationCore` dep

`MonetizationStateStore = AdGateStateStore` requires `public import MonetizationCore` in Persistence. The new SudokuKit Package.swift adds it to the Persistence target. This is mostly a leaf transitive — `MonetizationCore` has no third-party imports (foundations.md §9.1) so the dep direction stays clean.

### SudokuUI target gains a `MonetizationCore` dep (only)

`RouteFactory` stores `any AdProvider` / `any IAPClient` / `AdGate` to feed destination Views in v2.3.4-6. The protocol shapes all live in `MonetizationCore` — SudokuUI does NOT depend on `AdsAdMob` / `IAPStoreKit2` / `MonetizationTesting`. That isolation is the whole point of foundations.md §9.1: only `AppComposition` (the composition root) touches the concrete-impl targets. Verified by inspection: `SudokuUI/RouteFactory.swift` imports `MonetizationCore` only.

### App target explicit links to AdsAdMob + IAPStoreKit2 in Tuist

`Project.swift` adds `.package(product: "MonetizationCore")` / `"AdsAdMob"` / `"IAPStoreKit2"` to the App target. SwiftPM transitively pulls these via `AppComposition`, but Tuist+Xcode sometimes need the explicit App-target link so the `.xcframework` slices land in the bundle (the AdMob SDK ships as an xcframework). Per the brief, did not run `tuist generate` — Leader will.

## §未決 (Open questions) — both resolved

### #1 RouteFactory's `AnyView` return tradeoff

**Resolved**: chose `AnyView` over an `associatedtype Destination: View`. Reasoning logged under §設計決定 above. The cost (a single AnyView wrap per route push) is dwarfed by the API ergonomics win (RootView and AppComposition stay non-generic; tests don't have to thread a Destination type through fixtures).

If a future profile shows SwiftUI diff churn driven by AnyView identity at the navigation boundary, the migration path is local: introduce `RouteFactory.view<V: View>(for:) -> V` with per-case opaque returns, fix up `NavigationStackHost.destination` to accept that, and the call sites stay identical. Not worth doing speculatively.

### #2 Sidebar / HomeView card "present native modal" action source

**Resolved**: stays as direct `GameCenterDashboard.present()` calls on the originating View. `RouteFactory` only handles route-pushed destination Views, not modal presentation. v2.3.4-6 will read `adProvider` / `iapClient` / `adGate` directly from `AppComposition` inside the destination Views that need them (HomeView's banner card, SettingsView's IAP rows). RootView remains free of monetization-dep imports.

This boundary is documented in `AppComposition.swift`'s property comment block (lines beginning "v2 monetization deps. v2.3.4-6 read these directly from individual Views…").

## §Hand-off to v2.3.4-6 (Part B)

- `composition.adProvider` / `iapClient` / `adGate` are wired and accessible.
- `LiveRouteFactory` holds the same trio internally — destination views (DailyHubView, PracticeHubView, BoardLoaderView, CompletionView, SettingsView) can be extended to take them as init args in their respective steps without touching `RootView`.
- The banner-slot work (v2.3.4 HomeView, v2.3.5 BoardView) will need to flow `adProvider` + `adGate` into HomeView / BoardView. Two ergonomic options for Part B's developer to consider: (a) extend `LiveRouteFactory` to construct the views with those deps pre-bound (cleanest — keeps the dep web at the factory), or (b) read them inline from `@Environment(AppComposition.self)` once we install a SwiftUI environment value (slightly invasive — adds an Environment key, but matches SwiftUI conventions for app-wide singletons).
- The Settings IAP rows (v2.3.6) likely want option (a) — `SettingsViewModel(persistence:, iapClient:)` extension — because the row's behavior is driven by the VM, not the View.

Recommended path for Part B: stick with constructor injection in `LiveRouteFactory`. Avoids global environment state, keeps tests reflecting reality.
