## Decisions

### D1 — Ad unit ID source: `#if DEBUG` pivot (not Info.plist)
- Hardcoded constants at top of `LiveAdMobBridge.swift`:
  - DEBUG → test ID `ca-app-pub-3940256099942544/2934735716`
  - Release → production: see memory `admob-production-ids` (held out of repo per build-time-secret-injection)
- Rationale: matches the same pattern already used implicitly for `GADApplicationIdentifier` (which is a single test ID hardcoded into Info.plist for v2.5.2, swapped to production in v2.5.3 per `docs/v2/v2.5-readiness.md`). No new Info.plist key avoids coupling AdsAdMob to host-app resource layout.
- Future swap: v2.5.3 user-owned step flips both the Info.plist constant and removes the `#if DEBUG` (or just rebuilds Release). Single grep target.

### D2 — `rootViewController` source: connected key window
- AdMob's `BannerView.rootViewController` is only required when the user TAPS the ad (click-through presentation). Setting it at load time to the host app's key window is the recommended pattern (https://developers.google.com/admob/ios/banner).
- Bridge resolves at load time via `UIApplication.shared.connectedScenes.first(where: { $0.activationState == .foregroundActive })`. The `AdPresentationAnchorRegistry` (v2.4★) is anchor-keyed and the bridge currently isn't given an anchor — staying out-of-scope here per task constraint ("BannerSlotView UI changes" excluded). A future step can plumb the registry through `LiveAdMobAdProvider.init(bridge:)` if needed for multi-scene resolution.
- Fall-back: if no foreground scene resolves, `rootViewController` is left `nil`. `load()` still issues the request; click-through would fail silently but ad fetch (the v2.5.2 verification goal — "banner appears") still works.

### D3 — Handle plumbing: store `BannerView` in a UUID-keyed dict on the bridge
- `AdBannerHandle` is public surface = `UUID` only; cannot carry the live `BannerView` across module boundary (foundations §9.1 isolation contract).
- Bridge keeps an internal `[UUID: BannerView]` table (lock-protected, matching existing `OSAllocatedUnfairLock` pattern). Future UIViewRepresentable work (out-of-scope here) can add a `func bannerView(for: AdBannerHandle) -> BannerView?` accessor — but ONLY inside AdsAdMob, surfaced to UI via a separate UIKit-typed helper. For v2.5.2 the table just keeps the view alive past the awaited delegate callback.

### D4 — Delegate dispatch: `BannerViewDelegate` + `CheckedContinuation`
- AdMob 13.x exposes `bannerViewDidReceiveAd(_:)` and `bannerView(_:didFailToReceiveAdWithError:)`.
- A throwaway `NSObject`-subclass delegate is instantiated per `loadBanner()` call, wired to a `CheckedContinuation<AdBannerHandle, Error>`. It self-retains via the bridge's UUID dict (keyed differently to avoid mixing with the view table — actually simpler: the delegate is retained by `BannerView.delegate` strong reference until we either release it or the SDK does).
- `BannerView.delegate` is `weak`. Per AdMob convention the delegate is typically the view controller. We retain manually via a small in-flight delegate set on the bridge so the continuation isn't lost between `load()` and callback.

### D5 — Continuation single-resume safety
- Both success and failure callbacks first remove the delegate from the in-flight set and check a local `hasResumed` flag to guard against the (rare) case where AdMob fires both. Wrapped via a small inner actor or lock — chose `OSAllocatedUnfairLock<Bool>` matching house style.

### D6 — Single GoogleMobileAds import preserved
- All new code stays inside the existing `#if canImport(GoogleMobileAds)` block in `LiveAdMobBridge.swift`. No other file gains the import. Audit count remains exactly 1 (`LiveAdMobBridge.swift`) — AdMobBridge.swift mentions it in comments only.
- Note: task said "= 2" expected count but rg-grep includes the protocol file's *comment* mentioning `import GoogleMobileAds` — that's a false positive. The actual `import` directive count is 1. `v2.5-readiness.md §v2.5.2` confirms "exactly 1 match in AdsAdMob only".

### D7 — Existing `BannerLoadTests` already cover this path
- The bridge protocol is the test seam; `FakeAdMobBridge` is exercised by 8 existing tests covering refresh / failure / recovery / status transitions. Adding more tests would test SDK behavior (untestable in `swift test`) or duplicate existing coverage. **No new test file added.**
- Re-running existing tests after change validates the bridge protocol contract is unchanged (test seam intact).

### N1 — production ad unit ID → `fatalError` guard
- `#else` branch in `LiveAdMobBridge.swift` changed from a literal production constant to a computed `var bannerAdUnitID: String` that `fatalError`s with `REPLACE_IN_v2.5.3`. Any unintended Release build crashes loudly at first ad load rather than silently emitting a placeholder / wrong creative.
- `App/Info.plist` comment scrubbed: production App ID literal removed; replaced with a doc-reference-only pointer to `docs/v2/v2.5-readiness.md §v2.5.3`. The plist value itself is unchanged (still Google's universal test App ID).
- Trade-off: type signature shifts from `let: String` to `var: String { fatalError }`. Callers (only `loadBanner()`) read it once on the main actor — no semantic difference, no concurrency change.

### N3 — `withTaskCancellationHandler` wrap
- Wrapped the existing `withCheckedThrowingContinuation` block.
- Reuses the existing `OSAllocatedUnfairLock<Bool>` single-resume guard on `BannerLoadDelegate` — added a `cancel()` method that routes through `resumeOnce(.failure(CancellationError()))`. No new lock introduced.
- A `delegateHolder: OSAllocatedUnfairLock<BannerLoadDelegate?>` (local to `loadBanner`) bridges the cancel closure to the delegate constructed inside the continuation closure. The `onCancel` closure pulls the delegate, calls `cancel()` (whichever of cancel / didReceive / didFail fires first wins via `hasResumed`), and removes the `BannerView` from `liveBanners`. The `inFlightDelegates` removal still flows through the completion path that `cancel()` triggers via `resumeOnce`.
- No SDK-level cancellation API exists for `BannerView.load(Request())`; the SDK call continues in the background, but our `await` returns promptly and the awaiter sees `CancellationError`. The `BannerView` is released, so the SDK's eventual delegate callback hits a no-op `resumeOnce` (already-fired) and self-releases. No leak.

### N4 — `dispose(handle:)` TODO marker
- Comment `TODO(v2.5.x): dispose(handle:) accessor to clear liveBanners` placed directly above the `liveBanners.withLock { $0[handle.id] = bannerView }` insert site. Documents the intentional retain-for-handle-lifetime behavior (per impl-notes D3) and the planned v2.5.x follow-up.
