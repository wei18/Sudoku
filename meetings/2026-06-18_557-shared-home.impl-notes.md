## 設計決定 (Design decisions)

- **ATTPrimerSheet relocation** — Spec §Target shapes says `makeGameApp` applies universal `.attPrimerSheet(deps.attPrimer)`. The modifier lives in `SudokuUI` (internal) and GameAppKit cannot import SudokuUI (cycle). Decision: **move `ATTPrimerSheet` + `attPrimerSheet(_:)` extension to `MonetizationUI`** (where `ATTPrimerCoordinator` already lives). All string keys resolve via `Bundle.main` so they work cross-game as long as each app's xcstrings ships the keys — Sudoku already does. The view is made `public` in MonetizationUI. `MakeGameApp.swift` applies `.attPrimerSheet(deps.attPrimer)` on the `GameRoot` wrapper view (same level as the `.alert`). The existing `SudokuUI/Monetization/ATTPrimerSheet.swift` file is deleted.

- **`GameHomeView` placement** — The spec says `makeGameApp` builds `GameHomeView` as the `rootContent` of `GameRoot`. `GameHomeView` is placed in `GameAppKit` (the shared composition layer, deps allowed). It consumes `GameDeps` directly and owns: HomeScreen + ResumePill header + BannerSlotView banner + nav title + bootstrap `.task`. The `.attPrimerSheet` and GC-signed-out `.alert` are applied outside (on `GameRoot`'s returned view) by `makeGameApp` — not inside `GameHomeView` itself. This mirrors the current `RootView` structure where alert + attPrimerSheet are on the outer view, not on `HomeView`.

- **`GameHomeViewModel<Route>` placement** — Generic VM in `GameAppKit`. Identical logic to `SudokuUI.HomeViewModel` with `Route` as generic parameter replacing hardcoded `AppRoute`. The `RoutePath<Route>` path struct is already in `GameShellUI`, which `GameAppKit` imports. Subtitle copy + route mapping injected via `GameConfig` fields (new `homeModes: [HomeMode: HomeModeContent]` struct). The `HomeViewModel` typealias in `SudokuUI/Home/HomeViewModel.swift` is deleted (no longer needed).

- **`GameConfig` extension for home content** — Added `HomeModeContent` struct carrying `subtitleKey: LocalizedStringKey` + `route: Route?` (nil = leaderboard side-effect). Added `homeModes: [HomeMode: HomeModeContent]` to `GameConfig`. Sudoku's subtitle copy moves from `HomeViewModel.subtitleKey` extension to `Live.swift`'s config literal. The a11y namespace is not modelled separately — the `cardAccessibilityIdentifier` closure is kept nil (Sudoku never set accessibility identifiers; only MS did, and MS isn't migrated here).

- **`sidebarItems` from `GameConfig`** — Current `Live.swift` sets `sidebarItems: []` with comment "Sudoku builds sidebar from live HomeViewModel inside RootView". After #557, `GameHomeView` builds the sidebar items from `viewModel.modeItems` and the spec calls for them to come from `config.sidebarItems`. Rather than duplicating, `makeGameApp` derives sidebar items from the home mode config and passes them as `HomeModeItem.sidebarItems(from:)` to `GameRoot`. The existing empty `[]` config field is updated to carry the derived items. This ensures sidebar + home cards stay in sync from one source.

- **`wired.view` as mount point** — `AppComposition.rootView` currently constructs `RootView(...)`. After #557, `rootView` returns `wired.view` (the `AnyView` from `makeGameAppWithDeps`). The `bootMonetization()` call on `AppComposition.rootView` is removed because `makeGameApp` already calls `bootMonetization(adProvider:telemetry:)` via `MakeGameApp+Helpers`. The `.sudokuCell` environment and the `DEBUG` near-win modifier still need to be applied — these are kept in `AppComposition.rootView`.

- **Snapshot tests migration** — `RootViewTests` constructs `RootView` directly. `HomeViewTests` constructs `HomeView` directly. Both must migrate to construct `GameHomeView` equivalents. Since these tests live in `SudokuUITests` which imports `@testable import SudokuUI`, and `GameHomeView` is in `GameAppKit`, the tests need `import GameAppKit`. The snapshot PNGs must stay byte-identical — `GameHomeView` renders identically to the `RootView` → `HomeView` chain for the tested surfaces (no resume, with resume, iPhone/Mac light). The behavioral tests (`selectDailyAppendsDailyRoute` etc.) migrate to use `GameHomeViewModel`.

## 偏離 (Deviations)

- **`monetizationController` param dropped from `GameHomeView`** — The current `HomeView` accepts a `MonetizationStateController?` and calls `await controller.bootstrap()` in `.task`. In `makeGameApp`, `MonetizationStateController.startListeningForLifetimeOfApp()` is called once at wiring time, and the monetization boot happens via the `bootMonetization` helper (called in the view's `.onAppear`). Since `GameRoot` already has the `.onAppear { Task { await bootMonetization(...) } }` applied in `makeGameApp`, the `GameHomeView` does NOT need to call `bootstrap()` again. The `MonetizationStateController` is not passed to `GameHomeView`. Deviation from current `HomeView.task` behavior: both call the same underlying `MonetizationBootCoordinator.boot()` — the idempotency is handled by `makeGameApp`'s single wiring, not by `GameHomeView`'s `.task`.

- **`a11y namespace` not added to `GameConfig`** — Spec mentions "a11y namespace" per-game. Checked: Sudoku's `HomeView` sets no `accessibilityIdentifier` on cards. The `cardAccessibilityIdentifier` closure in `HomeScreen` defaults to `{ _ in nil }`. Not needed for Sudoku migration; adding it to `GameConfig` would add API surface with no consumer. Deferred.

## 折衷 (Tradeoffs)

- **Move ATTPrimerSheet to MonetizationUI vs. use @ViewBuilder slot** — Considered: (A) keep sheet in SudokuUI, have `makeGameApp` accept a `@ViewBuilder attPrimerContent` closure (adds API complexity); (B) move `ATTPrimerSheet` to `MonetizationUI` as a `public` view (sheet strings resolve from Bundle.main = app bundle, correct). Picked **B**: cleaner, aligns with where `ATTPrimerCoordinator` lives, strings already use `LocalizedStringKey` + Bundle.main resolution. Rejected A: adds closure churn with no benefit.

- **ATTPrimerSheet: theme colors INJECTED as params** — `MonetizationUI` does not depend on `GameShellKit` (would add a new dep edge), so it cannot read `\.theme` directly. **Initial (rejected by review) approach** used SwiftUI system styles (`.tint`/`.primary`/`.secondary`), which rendered the sheet system-blue instead of Sudoku sage-green — a visual regression. **Final approach** (review Finding 3): `ATTPrimerSheet.init` + the `.attPrimerSheet(...)` extension take four `Color` params (accent / primary-text / secondary-text / background) with system-style defaults; `makeGameApp` passes `config.theme.{accent.primary, text.primary, text.secondary, surface.background}.resolved`. This mirrors `BannerSlotView`'s injected-color pattern and restores byte-for-byte the pre-#557 `SudokuUI.ATTPrimerSheet` theming. Default params keep previews / un-themed hosts sensible.

- **`HomeModeContent` shape** — Considered: `[HomeMode: LocalizedStringKey]` (subtitle only) + separate route mapping `[HomeMode: Route?]`. Picked: `HomeModeContent(subtitleKey:, route:)` combined so each mode's presentation data is co-located. The `route: Route?` nil-means-leaderboard convention is clear and safe.
