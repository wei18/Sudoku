## Rollback plan

Two kill-switches are available without a code change:

- **Banner crash on real device**: ASC Phased Release pause; rotate AdMob
  ad unit to one that returns 0% fill — the banner slot will show
  `Ad unavailable` placeholder instead of a crash. If `LiveAdMobBridge`
  itself crashes, ship a hotfix that throws from `start()` (banner slot
  honestly renders `.failed`).
- **IAP failures**: rotate the ASC product to `Removed from Sale` — the
  Settings + HomeView Remove Ads CTAs will see an empty `availableProducts`
  list and fall back to the `"$2.99"` placeholder, but `purchase()` will
  return `.failed` honestly with a toast.

If a deeper kill-switch is needed, the cleanest seam is
`MonetizationStateController` — add a `disabled: Bool` flag read from
remote config / env var that short-circuits `purchaseRemoveAds()` and
flips `hasPurchasedRemoveAds = false` so the Settings rows disappear.
Tracked as a future spike, not yet built.

## App Privacy questionnaire (manual)

> **Why manual?** The official App Store Connect REST API has no endpoint
> for the App Privacy questionnaire. Fastlane's `upload_app_privacy_details_to_app_store`
> uses Apple-ID session cookies against undocumented endpoints — that's a
> security downgrade vs our `.p8` JWT auth, so we fill the form by hand.
> Investigation log: `meetings/2026-05-23_asc-app-privacy-api-research.impl-notes.md`.

### Navigation

1. Sign in to App Store Connect → **Apps** → **Sudoku** (app ID `6771248206`).
2. Left sidebar → **App Privacy** (隱私權).
3. First time only: click **Get Started** (開始使用). On subsequent edits,
   click **Edit** (編輯) next to "Data Types" (資料類型).
4. URL pattern: `https://appstoreconnect.apple.com/apps/6771248206/distribution/privacy`

### Step 1 — Data Collection declaration

| Question (en) | Question (zh-Hant) | Answer |
|---|---|---|
| Do you or your third-party partners collect data from this app? | 您或您的第三方合作夥伴是否會從此 App 收集資料？ | **Yes, we collect data from this app** / 「是，我們會從此 App 收集資料」 |

Rationale: AdMob SDK collects ad-targeting signals on our behalf — see
`PrivacyInfo.xcprivacy` `NSPrivacyTracking = true`.

### Step 2 — Data Types (must match PrivacyInfo.xcprivacy 1:1)

PrivacyInfo declares **one** entry: `NSPrivacyCollectedDataTypeOtherUsageData`,
not linked, used for tracking, purpose = Third-Party Advertising.

In the ASC questionnaire, tick **only** the following:

| ASC category (en) | ASC category (zh-Hant) | Tick? | Maps to xcprivacy |
|---|---|---|---|
| Usage Data → **Other Usage Data** | 使用資料 → **其他使用資料** | ✅ | `NSPrivacyCollectedDataTypeOtherUsageData` |
| (all other categories) | (其他全部類別) | ☐ | not declared |

Click **Next** / 「下一步」.

### Step 3 — Per-type details for "Other Usage Data"

For the single ticked entry, answer the three follow-up questions:

| Question (en) | Question (zh-Hant) | Answer | Maps to |
|---|---|---|---|
| Is this data used to track users? | 是否使用此資料來追蹤使用者？ | **Yes** / 「是」 | `NSPrivacyCollectedDataTypeTracking = true` |
| Is this data linked to the user's identity? | 是否將此資料連結至使用者的身分？ | **No** / 「否」 | `NSPrivacyCollectedDataTypeLinked = false` |
| What is the data used for? (multi-select) | 此資料的用途為何？（可複選） | **Third-Party Advertising** only / 僅勾「第三方廣告」 | `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising` |

Do **not** tick: Analytics / App Functionality / Developer's Advertising or
Marketing / Product Personalization / Other Purposes — those are not
declared in `PrivacyInfo.xcprivacy` and ticking them would create an
inconsistency at App Review.

### Step 4 — Privacy Choices URL (optional, leave blank)

Already covered by the in-app ATT prompt + UMP consent UI. Leave **User
Privacy Choices URL** empty.

### Step 5 — Publish

1. Click **Save** (儲存) → **Publish** (發佈) at top-right.
2. Confirm the legal attestation dialog ("Responses are accurate, comply
   with App Review Guidelines and applicable law, will be promptly updated
   if practices change").
