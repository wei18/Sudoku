## §設計決定 (Design decisions)

### Bridge seam shape (lifted v2.1.2's `StoreKitBridge` to v2.1.1)

Dispatch brief for v2.1.1 said "test `IAPProductMapper.map(...)` with mock `StoreKit.Product` fixtures". This is **infeasible** — `StoreKit.Product` has no public init; you cannot construct an instance in a unit test, only fetch one from the App Store.

Resolution: introduce the `StoreKitBridge` seam from v2.1.2 **in v2.1.1**, so the mapper operates on a fake-able `BridgeProduct` snapshot type instead of `StoreKit.Product` directly. The seam was going to land in v2.1.2 anyway — pulling it forward by one step removes the "test the untestable" footgun. Net: one bridge protocol covers both steps; no duplication.

Internal types added:
- `StoreKitBridge` (protocol) — `products(for:)`, `currentEntitlements()`, `purchase(productId:)`, `sync()`, `transactionUpdates()`
- `BridgeProduct` — Sendable snapshot of `StoreKit.Product` (id, displayName, displayPrice)
- `BridgePurchaseOutcome` — mirrors `Product.PurchaseResult` plus a `.failed(reason:)` case for thrown / unverified outcomes
- `BridgeTransactionEvent` — `.purchased(productId:)` / `.revoked(productId:)`, derived from verified `Transaction` snapshots

Production wiring: `LiveStoreKitBridge` (struct, `internal`) wraps the real StoreKit globals. `LiveStoreKit2IAPClient` only sees the bridge protocol — zero `import StoreKit` in the client file.

### `LiveStoreKit2IAPClient` shape
- `public actor` — needs cross-task safety for the small bit of state passed into the bridge calls. Brief mandated actor.
- Public `init()` for production (wires `LiveStoreKitBridge()` + `IAPProductIDs.all`).
- Internal `init(bridge:knownProductIds:)` for tests; injects fake + ID set.
- `purchaseUpdates()` is `nonisolated` (matches `FakeIAPClient` convention from v2.0 `MonetizationTesting`) — the AsyncStream is built per-call from the bridge stream; no actor state touched.

### `IAPProductMapper` purity
Pure `static func map(_ product: BridgeProduct, isPurchased: Bool) -> IAPProduct`. 18 lines. Independently unit-tested via 4 tests with no client / bridge involvement, matching the brief's "Pure function, independently testable" requirement.

### `IAPProductIDs`
- `internal enum IAPProductIDs` with `removeAds = "com.wei18.sudoku.iap.remove_ads"` per brief.
- Added `static let all: Set<String>` so `LiveStoreKit2IAPClient` doesn't need to hard-code the set inline — single source of truth. Pure additive, alphabetically ordered for diff stability.

### `purchase()` post-success product refetch
Brief says `IAPPurchaseResult.success(IAPProduct)` — the returned product should carry the freshly-purchased entitlement. Implementation refetches via `bridge.products(for: [id])` + `bridge.currentEntitlements()` after a successful `BridgePurchaseOutcome.success`. Costs one extra bridge round-trip; gains correct `isPurchased = true` stamp without trusting the client's in-memory state.

Edge: if the catalog refetch returns empty (rare — would mean ASC removed the product mid-purchase), synthesize a minimal `IAPProduct` carrying just `id` + `isPurchased: true` so `.success` still carries a usable handle rather than degrading to `.failed`.

### `restorePurchases()` flow
1. `bridge.sync()` (= `AppStore.sync()`).
2. `bridge.currentEntitlements()` (= iterate `Transaction.currentEntitlements`, filter `revocationDate == nil`).
3. `bridge.products(for: entitled)` to fetch display fields for entitled IDs.
4. Map each to `IAPProduct(isPurchased: true)`.

Returns `[]` if no entitlements — caller (Settings → Restore Purchases UI) shows "Nothing to restore". `sync` is always called even when entitlements are empty — that's the contract of Restore Purchases, sync first then read.

### `purchaseUpdates()` pipe
- `LiveStoreKitBridge.transactionUpdates()` builds an `AsyncStream` backed by a detached background `Task` iterating `Transaction.updates`, classifying each verified transaction as `.purchased` or `.revoked` based on `revocationDate`, then calling `transaction.finish()`.
- Unverified transactions are dropped — we never adjust entitlement state on an untrusted signal.
- `LiveStoreKit2IAPClient.purchaseUpdates()` is a 1:1 enum re-shape from `BridgeTransactionEvent` to `IAPPurchaseEvent`. Cancellation cascades via `continuation.onTermination` → `task.cancel()`.

### `FakeStoreKitBridge` (test target)
- `final class @unchecked Sendable` + `NSLock`-guarded state, not an `actor`. Reason: tests are synchronous bodies that interleave scripting calls (`setProduct`, `emit`) with `await`s on the client; an actor-typed fake would force every scripting call to be `await`-ed, making test code noisy. The lock is tiny and only contended in the AsyncStream emission path. Pattern mirrors the bridge-pattern fakes already common in Apple SDK testing (see also: SudokuKit `FakeGameCenterController`).
- Scripting API mirrors `FakeIAPClient` from v2.0 `MonetizationTesting`: `setProduct`, `setEntitlements`, `setPurchaseOutcome`, `setProductsError`, `setPurchaseError`, `setSyncError`, `emit`, `finishUpdates`. Call counters for `productsCallCount`, `purchaseCallCount`, etc.
- Lives in the test target (not `MonetizationTesting`) — it's a bridge-internal test seam, not a public-API fake. Keeping it internal means `MonetizationTesting` doesn't need to import `IAPStoreKit2`.

## §偏離 (Deviations from design / brief)

1. **Bridge seam introduced in v2.1.1, not v2.1.2** — see §設計決定 above. Net effect: v2.1.1 ships with `StoreKitBridge` + `BridgeProduct` already in place; v2.1.2 just adds the purchase / restore / updates methods on the existing protocol. No re-architecture between steps.
2. **`IAPProductMapper.map` takes `BridgeProduct`, not `StoreKit.Product`** — necessary consequence of #1. Mapper is still pure, still independently testable, still 18 lines.
3. **`Product.products(for:)` called twice in `purchase()` success path** — once via the bridge's purchase outcome, once via the client to refetch the post-purchase entitlement. The second call is the price of returning an authoritative `isPurchased: true` in the result. Acceptable; alternative was trusting the client's in-memory entitlement guess.
4. **Module placeholder `IAPStoreKit2.swift` + matching test removed** — v2.0 ModuleAnchor + `IAPStoreKit2ModuleTests.moduleLinks` test. Both replaced by real code. Test count delta: -1 placeholder + 22 real = +21 net new IAPStoreKit2 tests.

## §折衷 (Trade-offs)

1. **No `.storekit` configuration file** — brief mentions it as a testing tip, but unit tests bypass StoreKit entirely via `FakeStoreKitBridge`. The `.storekit` file would only matter for integration / device-runtime testing, which is out of scope for v2.1.
2. **`@unchecked Sendable` on `FakeStoreKitBridge`** — same trade-off the v2.0 notes called out for `AdPresentationAnchor`. `NSLock` guards all mutable state; the invariant is documented at the type declaration.
3. **`internal import` everywhere in IAPStoreKit2 sources** — per `InternalImportsByDefault` upcoming feature already on in `Package.swift`. Only `LiveStoreKit2IAPClient` uses `public import MonetizationCore` because its public protocol conformance to `IAPClient` requires `IAPProduct` / `IAPPurchaseResult` / `IAPPurchaseEvent` to be publicly visible from the module.
4. **No ATT prompt** — brief explicitly excluded. StoreKit does not require ATT; only ad tracking does. ATT lands in v2.2 AdMob.

## §未決 (Open questions for Leader)

1. **`recordRefund()` wiring** (v2.0-inherited §未決 #2): `purchaseUpdates()` now correctly emits `.revoked` when a verified `Transaction.updates` event carries a non-nil `revocationDate` (or family-share entitlement is dropped). v2.1 does NOT touch `AdGate` — that wiring is v2.3's job. Status: **deferred to v2.3 as expected; no v2.1 action needed**.

2. **`StoreKit.Product.products(for:)` empty-result behavior** — when ASC has no record of the ID (typo, wrong bundle, sandbox config drift), `products(for:)` returns `[]` rather than throwing. Live bridge currently treats this as "no products available" (returns `[]` from `availableProducts()`); `purchase()` returns `.failed(reason: "product not found: …")`. Reasonable default but worth documenting in design.md §How.IAP for the failure-mode catalog. **Non-blocking.**

3. **`@unknown default` on `Product.PurchaseResult`** — Apple may add a new case in a future iOS release; we return `.failed(reason: "unknown PurchaseResult case")`. If Apple adds something we want to handle differently (e.g. a `.requiresAuthentication`-style case), the live bridge needs updating. Compiler will not warn — only runtime behavior changes. **Non-blocking; flag for SDK upgrade reviews.**

4. **`Task.detached` priority on transaction observer** — `LiveStoreKitBridge.transactionUpdates()` uses `.background`. Real-world consideration: a refund event arriving while the user is paying matters for UX latency. v2.3 wiring may want to upgrade to `.utility` or default. **Non-blocking; revisit during v2.3 integration.**
