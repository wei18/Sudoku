Denial path & opt-out (proposal §5.3 + §6)

If the user taps Don't Allow on the system prompt (S04), the grant is permanent until changed in Settings. We never re-fire the system prompt (it won't show). Instead: a gentle in-app explainer with a deep-link to Settings (iOS openNotificationSettingsURLString; macOS = textual, P12 gap), then re-ask only via the primer at a later genuine value moment. A Settings toggle lets an authorized user opt out, cancelling the scheduled reminders.

State machine, use cases & flow legend

The ReminderAuthStatus machine and the arrow language for the canvas above. Foundation types (ReminderKind, ReminderAuthStatus, NotificationAuthorizing + ReminderScheduler seams) merged in #318; the primer view (Phase 2) is the only net-new UI, living in SettingsUI so both Sudoku and Minesweeper render an identical sheet with injected copy.

ReminderAuthStatus machine

.notDetermined cold launch · no ask (S01)
value moment → soft primer (our UI, repeatable)
primer (S03) "Not now" loops back, no system prompt
accept primer → system one-shot requestAuthorization (S04)
.authorized schedule dailyReady · deliver (S05)
— or — "Don't Allow"
.denied explainer + Settings deep-link (S06)
Settings → Allow → next-foreground reconcile → .authorized
.provisional reserved · low-stakes (U3 comeback)

.provisional = quiet delivery, no upfront prompt; the notification "sells itself" and the user keeps/turns-off after a few deliveries. Reserved for U3, not used for U1 — a Daily nudge must be visible, which provisional's silent delivery defeats (proposal §5.2). The provisional: Bool seam stays in the API for later A/B without an API change.
