## §設計決定 (Design decisions)

### Package layout
- 4 production targets (`MonetizationCore`, `AdsAdMob`, `IAPStoreKit2`, `MonetizationTesting`) + 4 matching test targets, mirroring SudokuKit's structure (one test target per production target).
- Swift settings shared via top-level `let swiftSettings`: `StrictConcurrency`, `ExistentialAny`, `InternalImportsByDefault` upcoming features + `swiftLanguageModes: [.v6]`. Identical to SudokuKit so behavior is consistent across packages.
- No external SPM deps in v2.0 — `dependencies: []`. AdMob comes in v2.2; pointfreeco/swift-snapshot-testing not needed (no UI / snapshots here).
- Placeholder `ModuleAnchor` per target: empty `internal enum` (not a `struct` — no instances should exist; `enum` is the conventional Swift "namespace / anchor" idiom and avoids implicit memberwise init). One file per target so `swift build` resolves a non-empty module.

### `AdProvider` / value types (§How.2)
- `AdBannerHandle.id` defaults via `init(id: UUID = UUID())` — design.md only specified `let id: UUID`, but a default arg is essential for ergonomic call-site construction in fakes / tests. Pure additive, no semantic drift.
- `AdBannerStatus` declared `Sendable, Equatable` per design; auto-synthesized works because `AdBannerHandle` is Equatable.
- All public types annotated `Sendable` explicitly even where the compiler could infer it — clarity over brevity.

### `IAPClient` / value types (§How.2)
- `IAPPurchaseResult` declared `Sendable` per design but **NOT** `Equatable` in the design source. Dispatch brief explicitly requested Equatable conformance for `.success` and `.failed` cases. Implemented via **manual `Equatable` conformance** (associated-value enums with mixed payload need manual `==` when one wants partial conformance; auto-synthesis would require every associated type to be Equatable — `IAPProduct` is already Equatable, `String` is Equatable, so auto-synth would in fact succeed for the entire enum). Went with **auto-synth `Equatable`** since all payloads are Equatable; simpler and covers all 4 cases.
- `IAPPurchaseEvent` likewise `Sendable, Equatable` (auto-synth, both cases have `String` payload).

### `AdPresentationAnchor` cross-platform shape
- Design said "iOS `UIWindow` / macOS `NSWindow` wrapper" with `#if canImport(UIKit)/AppKit)` fences.
- Dispatch brief specified: stored opaque `AnyHashable` ID + private extension lookup on host platform. Followed that contract:
  - Public surface: `public struct AdPresentationAnchor: Sendable { public let id: AnyHashable; public init(id: AnyHashable) }`.
  - Note: `AnyHashable` is **not** `Sendable` in the stdlib. To keep `AdPresentationAnchor: Sendable` under strict concurrency, I used `@unchecked Sendable` with a documented invariant ("callers must pass Sendable-compatible hashable values like UUID or a window identifier"). Annotated with a `// swiftlint:disable` style comment explaining why.
  - Platform-specific lookup extensions live in `#if canImport(UIKit)` / `#if canImport(AppKit)` blocks. v2.0 ships only the type; resolving `AnyHashable → UIWindow?` is deferred to v2.2.1 (AdsAdMob phase) where it's actually used — the design only mandates the *shape* here.
- Did NOT import UIKit/AppKit at module level — kept the type cross-platform pure for now.

### `AdGate` — `@MainActor actor` contradiction
- Design.md §How.3 shows `@MainActor public actor AdGate`. This is a Swift language contradiction: an `actor` already has its own isolation domain; `@MainActor` annotation on `actor` is rejected by the compiler.
- **Resolution**: chose `public actor AdGate` (drop `@MainActor`). Rationale:
  - The AdGate's job is state arbitration (read/write `AdGateState`), not UI work. Pure logic / store IO. Belongs on its own actor.
  - UI components that need its output (`BannerSlotView`) will `await` from MainActor, which is the normal Swift concurrency pattern.
  - If a future requirement forces MainActor (e.g., AdMob SDK must be called on main thread), wrap that in the `AdProvider` impl, not in `AdGate`.
- Flagged in §未決 below for Leader review.
- Removed `clock: any Clock<Duration>` injection from the brief — the actual frequency logic is **calendar-day based**, not duration-based. `Calendar` injection is sufficient for determinism; `Clock` is only useful if we were measuring intervals in seconds. Saved an unused dep.

### `AdGate` policy (§How.3)
Implementation reads pure design rules:
1. `hasPurchasedRemoveAds == true` → false (highest precedence).
2. `now < firstLaunchAt + 7 days` → false (grace).
3. `dismissedDate` is the same calendar day as `now` → false.
4. Otherwise → true.

Notes:
- `lastShownDate == today` did NOT cause `false`; design.md says "1/day max" but **banner is persistent within the day**, so once shown, keep showing until dismissed. The design comment "lastShownDate == today AND not yet dismissed → keep showing" means we only return false if dismissed. So the rule reduces to: dismissed today → false. lastShownDate is recorded but doesn't gate.
- "7 days" interpreted as `firstLaunchAt + 7 × 86400 seconds`. Day 0 is the launch day; day 7 boundary is exactly 7 days later. Test fixture: launch at midnight UTC, query at midnight + 7 days → true. At midnight + 6 days 23:59:59 → false.
- Calendar-day comparison uses `calendar.isDate(_:inSameDayAs:)`.

### State caching
- `AdGate` loads state on first access via internal `currentState() async throws` helper; subsequent reads use cached copy. Each mutation writes through to store. This matches "Internal state cached after first `loadState`" in the brief.
- Initial state if store throws / first ever launch: design says `firstLaunchAt: Date` is "set once on first ever launch". I interpret that as: the **store** is responsible for seeding on first ever launch. `AdGate` does not invent `firstLaunchAt`. Tests inject the store with a pre-seeded `firstLaunchAt`.

### Fakes (§How.2 / v2.0.3 / v2.0.5)
- `FakeAdProvider`: scriptable scenario list. `script(_:)` API takes an enum-like value (`ScriptedAdProviderState`) that bundles the initial status + behavior on next refresh. Kept simple: array of statuses to return in sequence on `bannerStatus` access.
- `FakeIAPClient`: `setProducts(_:)`, `setPurchaseResult(for:result:)`, `emit(_:)` for `purchaseUpdates`. Uses `AsyncStream.makeStream(of:)` and stores the continuation.
- `FakeAdGateStateStore`: actor with in-memory `var state: AdGateState?` + `loadCount` / `saveCount`. `script(_:)` seeds.

## §偏離 (Deviations from design / brief)

1. **`@MainActor actor AdGate` → `actor AdGate`** (drop `@MainActor`). See §設計決定 above. Design.md should be updated; flagged in §未決.
2. **`clock: any Clock<Duration>` dropped**. Replaced by `calendar: Calendar` only — `Clock` was never actually used in the gate logic, which is calendar-day based. Tests inject UTC calendar.
3. **`AdPresentationAnchor` uses `@unchecked Sendable`** — `AnyHashable` is not `Sendable`. Annotated with usage invariant. Alternative would be `id: UUID` directly, but that defeats the "opaque ID" requirement. Acceptable trade-off; the platform lookup extensions will validate at call-site.
4. **`IAPPurchaseResult: Equatable`** — auto-synthesized (not manual) because all payload types are Equatable. Equivalent semantics, less code.
5. **Placeholder type kind: `enum` not `struct`** — `internal enum ModuleAnchor {}` instead of `struct`. Better Swift idiom for empty namespaces (can't be instantiated by accident).

## §折衷 (Trade-offs)

1. **Test framework**: swift-testing (`import Testing`) per dispatch brief. No XCTest fallback. Matches SudokuKit baseline.
2. **No `Codable` round-trip via `JSONEncoder`** — request was "Codable / Equatable round-trip" for value types; `AdBannerHandle` is `Sendable + Equatable` (not declared Codable in design). Only `AdGateState` is `Codable`. Codable round-trip tests only on `AdGateState`. Other value types just get Equatable + Sendable assertions via generic helper (mirror of `assertSendable` pattern from Telemetry tests).
3. **`ProtocolShapeTests`**: protocol-witness style. Tests create a tiny in-test struct that conforms to `AdProvider` / `IAPClient` to verify the protocol shape compiles. Cheap, no behavioral testing of fakes (that's `FakeShapesTests`'s job).
4. **AdGate tests use UTC calendar** to eliminate DST + locale boundary noise. Real app will use `.current` calendar — that's a wiring concern, not a logic concern.

## §未決 (Open questions for Leader)

1. **`@MainActor actor AdGate` in design.md** — language-level invalid. Implementation dropped `@MainActor`. Either:
   - (a) design.md §How.3 should be amended to `public actor AdGate` (recommended), OR
   - (b) AdGate should be `@MainActor public final class` (loses Sendable-by-construction guarantee, gains MainActor isolation for UI direct calls).
   - Recommend (a). Leader to confirm and patch design.md in follow-up.

2. **`AdGate.recordPurchase()` precondition** — design says "permanently suppresses" / never reverses. Implementation flips to `true` and stays true. If a future refund flow needs to reverse (e.g. `purchaseUpdates` emits `.revoked`), there's no `recordRefund()` API in v2.0. Likely needed in v2.1 IAP wiring. Not blocking v2.0.

3. **`firstLaunchAt` seeding** — `AdGate` assumes store hands back a seeded `AdGateState`. The real `LiveMonetizationStateStore` (CloudKit-backed, v2.3) will need to seed `firstLaunchAt = .now` on its first `loadState()` if no record exists. v2.0 fake stores are pre-seeded by tests. Worth a comment in design.md §How.3.

4. **`AdPresentationAnchor` `@unchecked Sendable`** — accepted trade-off; if Leader prefers `id: UUID` (strict-checked Sendable, lose `AnyHashable` polymorphism), flag and I'll swap.
